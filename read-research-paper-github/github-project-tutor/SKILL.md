# GitHub Project Tutor

## 目标

把仓库还原成一条可验证的执行故事，让不了解项目的读者回答：项目解决什么问题、从哪里开始读、运行时先后发生什么、文件为何互相导入，以及每份数据在各模块间如何变化。

坚持“先证据，后解释”。不要把 README 宣传、命名猜测或常见框架习惯当成代码事实。

把“初学者”理解为尚不能自动区分 batch、token、channel、feature、sequence、rank、axis、broadcast、reshape 和 concat 的读者。不能只给 shape 结论；必须解释每个轴代表什么、哪个轴变化、为什么能执行以及最容易产生的误解。

不要把源码导读写成算子翻译。对关键设计继续追问：为什么需要这一步、它解决了什么约束、带来什么收益、牺牲了什么、替代实现是否等价、参数变化会怎样影响信息量/复杂度/显存，以及怎样验证理解正确。

## 工作流

### 1. 确定分析对象

1. 解析仓库、分支、tag、commit、子目录或具体文件链接。
2. 记录远程地址、默认分支和分析 commit。若用户指定 commit，保持不变。
3. 优先分析本地完整源码；只有网页内容时，获取仓库或相关源码文件。检查子模块和 Git LFS 指针。
4. 对超大 monorepo，先界定与用户问题相关的包，同时说明未覆盖范围。
5. 不默认执行不受信任代码。先静态检查；安装依赖、下载大型数据/权重或运行训练前，说明成本与风险并按当前授权规则处理。

### 2. 建立仓库地图

先读 README、依赖清单、构建文件、配置、启动脚本、示例和测试，再读核心源码。忽略 `.git`、缓存、构建产物、虚拟环境、大型数据与权重。

对本地仓库先找到本 Skill 所在目录，再运行：

```text
python <skill-dir>/scripts/repo_probe.py <repo-path> --format markdown
```

把输出当作索引，不当作最终结论。脚本只能发现静态候选；动态注册、反射、配置注入和运行时分支必须继续读源码确认。

产出三张地图：

- **职责地图**：目录/关键文件 → 功能 → 为何重要。
- **依赖地图**：本地模块 `A → B` 表示 A 导入或依赖 B；区分第三方依赖。
- **执行地图**：命令/入口 → 配置解析 → 对象构造 → 主循环 → 输出。

### 3. 找到真正入口

同时检查：

- `if __name__ == "__main__"`、CLI、console script、shell/Makefile/容器命令；
- `train.py`、`main.py`、`app.py`、`infer.py`、notebook、示例与测试；
- 配置中的 `_target_`、registry、factory、plugin、callback 和字符串类名；
- README 命令最终调用的实际函数。

不要只说“入口是 train.py”。继续追踪到第一个业务函数、数据构造、模型构造和循环函数。若训练、推理、评估、服务有不同入口，分别给出链路。

### 4. 逐层追踪调用与导入

对每条主链路记录：

```text
命令 → 文件:行 → 函数/类 → 下一个调用 → 产生或消费的数据
```

对每次跨文件跳转解释：

1. 谁导入谁；
2. 导入的符号是什么；
3. 在哪里实例化或调用；
4. 配置、参数或返回值如何连接两端；
5. 是否存在继承、回调、注册器或依赖注入造成的隐式连接。

优先展示影响主流程的关系，避免把全部 import 堆成噪声。复杂关系使用 Mermaid；节点写“文件/类 + 职责”，边写“imports / calls / returns / tensor”。

### 5. 深度学习项目执行端到端追踪

只要仓库包含模型训练、推理或数据管线，就必须完整读取 [deep-learning-tracing.md](references/deep-learning-tracing.md)、[beginner-explanation-standard.md](references/beginner-explanation-standard.md) 和 [design-rationale-interview.md](references/design-rationale-interview.md)，并执行其中检查表。至少覆盖：

1. 原始数据的物理含义、目录/标注格式和一个样本的结构；
2. `Dataset.__init__`、`__len__`、`__getitem__` 或等价输入管线；
3. transform/augment/tokenize/normalize，以及顺序为何重要；
4. sampler、collate、padding、mask 和 DataLoader 形成 batch 的过程；
5. 模型构造及配置值如何决定层数、通道、hidden size、head 数等；
6. `forward` 每个重要子模块、分支、残差、拼接和 reshape；
7. 输出如何进入 loss、各 loss 的输入和权重；
8. `zero_grad → forward → loss → backward → optimizer.step`，包括 scheduler、AMP、梯度累积/裁剪、分布式行为；
9. validation、inference、decode/post-process、metric 和最终输出；
10. 训练与推理链路的差异。

必须建立**张量形状账本**。每个关键站点写明：变量、语义、输入 shape、操作、输出 shape、dtype、device、代码证据和置信度。不能只写模型层名称。

对 tokenizer、embedding、padding、broadcast、concat/stack、reshape/view、flatten、permute/transpose、QKV 拆分/合并、残差和 loss flatten 等高认知负担操作，必须增加“你可能会问”小节。回答至少包含：

1. 一个具体疑问；
2. 变化前后每个轴的含义；
3. 哪个轴变了、哪个轴没变；
4. 一个带具体数值的最小例子；
5. 为什么数学上/框架上允许该操作；
6. 为什么项目选择该操作，而不只是“代码做了什么”；
7. 主要收益、代价和至少一种替代方案；
8. 与实际代码对应的证据。

例如不能只写“`[B,77,768]` 与 `[B,77,1280]` 拼成 `[B,77,2048]`”。必须解释沿 `dim=-1` 拼接意味着 token 数 77 不变、每个位置的特征被并排组合，并核对 batch 和 token 轴相同。

### 5.1 双轨讲解与问题密度

关键模块必须同时给出两条轨道：

- **数据轨**：输入 → 每个操作 → shape/dtype/device → 输出 → 下游；
- **设计轨**：问题 → 选择 → 数学机制 → 收益 → 代价 → 替代方案 → 边界条件。

按重要性控制面试追问数量：

- `P0` 主干且改变 token/分辨率/通道/语义或复杂度的操作：回答 4–8 个递进问题；
- `P1` 主干但 shape 不变的归一化、残差、门控等：回答 2–4 个问题；
- `P2` 明显样板代码：只说明作用与证据，避免机械制造问题。

问题必须带答案，不得只罗列题目。按“基础事实 → 数学机制 → 设计选择 → 权衡替代 → 故障诊断”递进，并优先生成能由当前仓库代码回答的问题。

### 6. 验证，而不是脑补

使用四级证据标签：

- `运行验证`：通过安全的最小执行、测试、hook 或日志观察到。
- `代码确认`：由函数签名、索引、算子和配置可静态推出。
- `合理推断`：代码不完整或依赖运行时值，但推断有明确依据。
- `暂未确认`：缺少数据、配置、权重、生成文件或动态路径。

静态推导 shape 时展示公式，例如：

```text
[B,C,H,W] --Conv2d(k,s,p)--> [B,Cout,floor((H+2p-k)/s)+1,floor((W+2p-k)/s)+1]
```

只有在安全且无需高成本资源时才做最小运行验证。可优先使用现有单元测试、shape assertion、示例输入或一个合成 batch。不要为了“验证”启动完整训练。若无法运行，明确缺失条件并继续完成静态分析。

### 7. 按初学者层次解释

先给全景，再逐步深入：

1. **30 秒说明**：输入、核心处理、输出、典型用途。
2. **推荐阅读顺序**：按执行链而非字母顺序列文件。
3. **项目地图**：只保留关键目录，并解释普通读者可暂时跳过什么。
4. **一次完整旅程**：用一个具体样本贯穿数据、模型、loss/输出。
5. **模型拆解**：每个模块先用白话说明目的，再讲实现和 shape。
6. **设计与面试追问**：解释关键实现的原因、等价写法、复杂度、代价和失败条件。
7. **关键机制专题**：注册器、配置系统、缓存、分布式等。
8. **复现指南**：命令、依赖、数据/权重和预期产物。
9. **不确定项与下一步**。

首次出现术语时解释含义。把抽象符号绑定到项目，例如“这里的 `B=8` 是每张 GPU 的 batch size，`N=196` 是 14×14 个图像 patch”。不要用另一个未解释的术语解释当前术语。

遵循“疑问驱动 + 轴驱动”写法。读者第一次遇到一个 shape 前，先定义全部符号；一个阶段结束时，给出“进入时是什么、离开时是什么、下一步谁会使用它”。若某个单词被拆成多个子词，明确说明增加的是 token 序列长度 `L`，而不是每个 token 的特征维度 `D`。若不同 tokenizer 产生不同 token 边界，不得声称它们逐位置对齐。

### 8. 输出与质量门槛

最终报告遵循 [report-template.md](references/report-template.md)。除非用户明确要求只在聊天回答，否则必须在当前工作区生成一个 UTF-8 Markdown 文件：

```text
reports/<owner>-<repository>-beginner-guide.md
```

同一仓库的后续修订更新同一个文件；只有用户要求保留历史版本时才另建带版本号的文件。超大项目可以增加 `reports/<owner>-<repository>-appendix.md`，但主报告必须能够独立阅读。

聊天回复只给结论摘要、覆盖边界和可点击文件链接，不把完整报告仅留在对话中。根据仓库规模调整篇幅，但不得省略入口、主调用链、逐阶段数据流、深度学习 shape（如适用）、证据和不确定项。

使用标准 Markdown 数学格式：行内 `$B\times L\times D$`，独立公式使用 `$$...$$`。不要用普通方括号模拟公式。生成目录、编号标题、术语表、统一符号表和证据索引。

提交前检查：

- 每条重要结论都有文件/行号或 commit 固定链接；
- 入口确实能连到数据、模型和输出，而不是停在包装脚本；
- import 图与 runtime call 图没有混为一谈；
- 每个重要 shape 变化解释了“怎么变”和“为什么变”；
- 每个 `P0` 模块解释了设计问题、选择、收益、代价、替代方案和边界条件；
- 每个高认知负担操作回答了至少一个真实的小白疑问；
- 面试问题有递进答案，不是脱离源码的通用题库；
- 复杂度结论写出自变量和缩放关系，例如 token 数改变如何影响 $O(N^2)$ attention；
- 关键可学习模块说明参数量怎样由通道、宽度、kernel/head 等配置决定；
- 构造函数参数、兼容字段和功能开关已追到 `forward`，未把“字段存在”误写成“功能生效”；
- 声称两个实现等价时写出成立条件，不把近似、特例或经验说成普遍等价；
- 每次 concat/stack 都写出 `dim`、兼容轴和变化轴；
- 每个 shape 都能追溯到变量、配置或算子，不把示意数字伪装成实际值；
- `view/reshape/permute/transpose/flatten/cat/stack/pad/pool` 前后维度完整；
- batch 维、序列维、通道维、空间维没有混淆；
- 配置默认值与用户实际命令覆盖值没有混淆；
- 无法确认的内容没有伪装成事实；
- 读者知道先看哪三个文件、如何继续验证。
- Markdown 文件存在、目录可导航、公式语法正确、链接使用固定 commit，并在最终回复中给出可点击路径。

## 适配非 Python 项目

保持同一方法：用语言对应的包清单、import/include、构建入口、路由、调用点和类型定义建立证据链。`repo_probe.py` 对 Python 最深入，对其他语言只提供仓库清单和入口候选，因此必须用语言原生搜索或静态工具补足。
