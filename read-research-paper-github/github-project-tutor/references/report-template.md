# GitHub 项目小白精读文件模板

## 目录

1. 文件交付规则
2. 报告目录结构
3. 章节写法
4. 表格与图示
5. 完稿检查

## 1. 文件交付规则

默认创建 `reports/<owner>-<repository>-beginner-guide.md`，使用 UTF-8、GitHub Flavored Markdown 和标准 LaTeX 数学语法。报告开头写仓库链接、branch/tag、commit、分析日期、分析方式和未覆盖范围。

主文件必须独立可读。不要把关键解释只放在聊天消息、临时日志或附录。超大仓库可把完整文件清单、次要调用链和超长 shape ledger 移入一个 appendix，但主报告保留结论和链接。

## 2. 报告目录结构

使用下面的编号骨架；无对应内容时明确写“仓库未包含”，不要静默删除章节。

```text
# <项目名>：面向初学者的源码精读

## 0. 如何阅读本报告
## 1. 分析基线与项目边界
## 2. 30 秒读懂项目
## 3. 先补齐的基础概念
## 4. 推荐源码阅读顺序
## 5. 项目目录与文件关系
## 6. 程序入口和完整调用链
## 7. 一份真实数据的端到端旅程
## 8. 核心模型逐模块拆解
## 9. 训练、推理、评估与输出闭环
## 10. 配置、权重和复现
## 11. 常见疑问与易错点
## 12. 证据等级、未确认项与下一步
## 附录 A. 完整张量形状账本
## 附录 B. 术语与符号表
## 附录 C. 代码证据索引
```

在 `0. 如何阅读` 中告诉读者第一次应该读哪些章节，哪些高级分支可以后读。

## 3. 章节写法

### 3.1 基础概念先于复杂流程

在第一次使用前解释 batch、sample、token、channel、sequence、feature、rank、axis、dtype、device。给统一符号表，例如：

| 符号 | 含义 | 本项目示例 |
|---|---|---|
| $B$ | batch size | 1 |
| $L$ | token 数 | 77 |
| $D$ | 每个 token 的特征宽度 | 4096 |

### 3.2 每个关键步骤使用同一模板

```text
### n.m <步骤名称>

本步解决什么问题？
输入从哪里来？
输入 shape 的每个轴是什么意思？
执行哪行代码/哪个算子？
哪个轴改变，哪个轴保持？
为什么数学上允许？
项目为什么选择这种实现？
它的收益、代价和替代方案是什么？
参数变化会怎样影响 shape、计算量、显存或信息保留？
输出交给谁？

#### 你可能会问：<具体疑问>
用最小数字例子回答，并指出常见错误理解。

#### 模拟面试追问
按“基础事实 → 数学机制 → 设计动机 → 权衡替代 → 边界/调试”给 2–8 个递进问题，并紧跟参考答案。关键可学习模块至少有一道参数量或参数缩放题；存在 flag 时核对它是否进入真实执行路径。
```

必要时继续拆成 `n.m.1`、`n.m.2`。不要让一个小节同时跨越 tokenizer、encoder、concat 和 model forward 四个认知步骤。

### 3.3 一个数据样本贯穿到底

选择一个实际输入或明确标为“示意”的输入。沿真实执行链记录：

```text
原始输入 → 解析/读取 → 单样本 → batch → device
→ 模型子模块 1 → 子模块 2 → loss/采样器
→ decode/post-process → 最终输出
```

每一步写变量名、shape、dtype、值域、device、代码位置和下游消费者。

### 3.4 模型模块不能只列名字

每个自定义模块回答：

- 为什么存在；
- 接收什么；
- 内部按什么顺序计算；
- 哪些参数决定 shape；
- 可学习参数量如何由关键尺寸决定；
- 输出什么并交给谁；
- 对整体任务有什么作用；
- 为什么选择当前算子/布局/参数，而不是常见替代方案；
- 计算量、显存、信息保留或优化稳定性的主要收益与代价；
- 实现等价需要满足什么条件；
- 哪些边界输入会失败，以及如何做最小验证；
- 构造函数中的模式/兼容开关是否真的被 forward 消费；
- 初学者最容易误解什么。

重复 block 可详细解释第一个，再列层号覆盖表，但所有改变 shape、分支或路由的层单独说明。

## 4. 表格与图示

至少包含：

1. 关键文件职责表；
2. 推荐阅读路线；
3. 维度符号表；
4. 逐阶段 shape ledger；
5. 模块覆盖表；
6. 证据等级表。

调用关系复杂时使用 Mermaid。边必须标注 `imports`、`calls`、`constructs` 或具体数据名，不能只画无含义箭头。

公式使用：

```markdown
行内：`$H\in\mathbb{R}^{B\times L\times D}$`

独立：
$$
H_{LG}=\operatorname{Concat}_{D}(H_L,H_G)
$$
```

代码事实使用 commit 固定链接。示意 token、数值或 shape 必须写“示意”；实际 tokenizer 结果必须由相同 tokenizer 和版本验证。

## 5. 完稿检查

- 报告文件存在并使用目标仓库名称；
- 有可导航目录和连续编号；
- 首次出现的术语和公式符号已定义；
- 每个 `cat/stack/reshape/permute/flatten/pad/broadcast` 都说明轴语义；
- 每个高认知负担步骤都有“你可能会问”；
- 每个 `P0` 关键模块都有带答案的递进面试追问链；
- 追问覆盖设计动机、收益代价、替代方案、复杂度和边界条件；
- 复杂度写明关于哪个 shape 变量缩放，避免只写“更快/更省显存”；
- 等价实现写明成立条件，经验判断与代码事实分开标记；
- 每个具体数值区分代码确认、运行验证和示意；
- Python 容器没有被误写成带 dtype/device 的张量；
- 模型输出的语义名称有训练目标或官方证据，不从推理公式猜测；
- 入口确实连到最终输出；
- 训练代码缺失时明确写缺失，不用论文常识补成仓库事实；
- 固定 commit 链接可访问；
- 复现步骤核对了文档下载文件名、代码实际路径、tokenizer 网络访问和 device；
- 主报告不依赖聊天上下文；
- 最终聊天回复包含文件链接、覆盖范围和验证状态。
